# Internationalisation (i18n)

## Model
- Languages provided via per-module lang files: `modules/{Name}/lang/en.php` and `ru.php`.
- UI strings are retrieved with the `__()` helper; no hardcoded text in views.
- Admin and frontend share the same lang resolution logic.

## Structure
- Keys are flat strings grouped by prefix (e.g., `dashboard.title`, `modules.actions.enable`).
- Both `en` and `ru` files must contain the same keys.
- Do not rename keys without updating all references.

## Usage
- In views: `__('key')`.
- In controllers: pass already-localised strings only when unavoidable; prefer to keep localisation in views.
- Place new strings in the appropriate module’s lang files, not in core.

## Locale switching
- Global locale set in admin Settings.
- Locale applies to both frontend and admin.
- Ensure critical pages (auth, settings) have complete translations in both languages.

## locale_mode — field visibility
Beyond the language used for output, `locale_mode` controls which input fields are shown in admin forms:

| Value | EN fields | RU fields | Required |
|-------|-----------|-----------|----------|
| `multi` | ✓ | ✓ | at least one title |
| `en` | ✓ | — | `title_en` |
| `ru` | — | ✓ | `title_ru` |

- Controllers read `$settings->get('locale_mode', 'multi')` and pass `localeMode` to the view.
- Views compute `$showEn = ($lm !== 'ru')` and `$showRu = ($lm !== 'en')`.
- Applied in: Articles, Article Categories, Gallery upload form.
- New bilingual admin forms must implement this pattern.

## Do/Don’t
- Do keep keys concise and descriptive.
- Do avoid concatenating translated fragments; use full phrases.
- Don’t mix languages in one locale file.
- Don’t fallback to hardcoded English in templates.***
